# 1. ReactとJavaScript
   ## 1.1 JavaScriptの歴史
   ### 初期の盛り上がりから「不遇の時代」へ
   - 1990年代
     - JavaScriptが初期の盛り上がりを見せていた時代。JavaScriptは視覚効果に使われていた。今も一部は利用されているが、この時代は過剰だった。その結果、重くダサいページが量産された
     - この時期、ブラウザ間の非互換問題や、度重なるセキュリティホールの報告も相まって、JavaScriptへの関心は急速に萎んでいった
   ### AjaxによるJavaScriptの復権
   - 2005年
     - Ajax（Asynchronous JavaScript + XML）の登場
     - Ajaxとは「ブラウザ上でデスクトップアプリライクなページを作成する技術」の総称
     - HTML、CSS、JavaScriptなど、ブラウザ標準の技術だけでリッチなコンテンツを作成できることから、Ajax技術は短期間で急速に普及した
     - 国際的な標準化団体であるEcma Internationalのもと、JavaScriptの標準化が進められ、ブラウザベンダーによる機能拡張合戦も落ち着いたことで、クロスブラウザ問題が軽減された
     - インターネットそのものの普及がセキュリティへの関心を高め、セキュリティホールが減少したのもこの時期
     - Ajax技術の普及は、JavaScriptの言語としての価値を見直すきっかけになった。AjaxではJavaScriptなしでは始まらない

  JavaScriptが本格的なアプリを開発するための道具と認識されたことで、コーディングの手法にも変化が現れる。従来のように、簡単であることをよしとするのではなく、大規模な開発にも耐えうる記法ー具体的にはオブジェクト指向なコーディングーが求められるようになった。その過程で、より熟練した開発者がJavaScriptの世界にも集い始め、開発のノウハウも蓄積され、さらに熟練した開発者が集まる、という好循環も生まれる。
  ### HTML5の時代へ
  HTML5の意義は、単なるマークアップ機能の進化ではない。ブラウザネイティブな機能だけでアプリを開発できる基盤ーJavaScript APIが整備されたことにある。
  |機能|概要|
  |--|--|
  |Geolocation API|ブラウザーの地理的な座標を取得|
  |Web Storage|ブラウザにデータを保存するストレージ|
  |File API|ローカルのファイルシステムを読み書き|
  |Canvas|JavaScriptで画像を描画|
  |Web Workers|JavaScriptのコードをバックグラウンドで実行|
  |WebSocket|サーバ、クライアント間での双方向通信|

  これらのAPIによって、JavaScriptだけでできることが格段に広がった。加えて、スマホやタブレットの普及によるRIA（Rich Internet Application）の衰退、SPA（Single Page Application）の流行なども、JavaScriptの存在感を高めることに拍車をかけた。

  ## 1.2 jQueryから新世代のライブラリへ
  JavaScriptを中心としたフロントエンド開発が盛んになってくると、関連するライブラリ、ツールの開発も活性化する。その中で、2000年代後半から10年近くにわたって、デファクトスタンダードの地位を守り続けたのがjQuery。
  ### jQueryの魅力と限界
  jQueryは基本的なページの操作からアニメーション、Ajax通信、標準JavaScriptの拡張など、JavaScriptにおけるUI開発を広くサポートする優れたライブラリ。簡単なアプリをごくシンプルなコードで実装できる手軽さは、現在でも魅力である。

  もっとも、そんなjQueryの魅力も、フロントエンド開発が高度化するにつれ、力不足の面も目立ってきた。たとえば、何かしらの入力をトリガーにデータを取得し、その結果をページに反映させる、といった処理も、jQueryでは
  - 入力値を文書ツリーから取得
  - Ajax通信に引き渡す
  - 取得した結果をページに埋め込む
  という操作が必要になる。JavaScriptの側では、入出力にあたって常にページの構造を意識しなければならない。
  このようなデータの受け渡しは面倒なもので、レイアウトとコードの混在は、アプリ全体の見通しを悪くする。日常的にページの操作を繰り返すSPAともなれば、jQueryで実装するのは現実的ではない。
  そこで求められたのが、ページとオブジェクト（JavaScript）との間を取り持つJavaScriptライブラリ（フレームワーク）である。アプリ全体を俯瞰し、ページに変化があればオブジェクトに反映させ、逆にオブジェクトが変化すればページに反映させるーそのための仕組みを提供する存在。これによって、アプリ開発者はテンプレート（HTML）、ロジック（JavaScript）それぞれの開発に集中できるので、コードの見通しも改善し、アプリの開発生産性、保守性が向上する。

  ## 1.3 主なJavaScriptフレームワーク
  jQueryの後発として登場してきた中で、現在特によく利用されているのは、以下のフレームワーク（ライブラリ）である。
  - React(https://ja.react.dev/)
    Metaが開発したフレームワークで、ビュー相当の機能を提供
  - Angular(https://angular.io/)
    Googleを中心に開発されているフルスタックフレームワーク
  - Vue.js(https://ja.vuejs.org/)
    ビューに特化したシンプルなフレームワーク

  大雑把にまとめると、フルスタック（全部入り）のAngularに対して、ビューに特化したReact、Vue.jsという分類になる。Angularはビューからサービスまで幅広くサポートした高機能フレームワークだが、導入のハードルが高いというデメリットがある。初期段階で学習すべき点が多い。また、既にあるアプリに対して後付けでAngularを導入するのは厄介である。最初からAngularを採用することを前提に、きちんと設計された環境でこそ、強みを発揮するフレームワークと言える。
  一方、Reactはビュー（見た目）の部分に特化したフレームワークなので、導入は簡単。学ぶことも限られる。Vue.jsに比べても、あらかじめ覚えなければならない構文が限られるので、最初に押さえなければならないことが少ない。
  いずれかがより優れているというわけではないので、状況に応じて強みを活かすと良い。

  ### 補足：ライブラリとフレームワーク
  - ライブラリ
    ユーザコードから呼び出されることを想定している（ライブラリ自体が自発的に何かをすることはない）。機械学習のためのライブラリ、メール送信のためのライブラリなど、ユーザコードからの指示を受けて処理を実施するのがライブラリである。
  - フレームワーク
    ユーザコードはフレームワークによって呼び出される。フレームワークがアプリのライフサイクル（初期化から終了までの流れ）を管理しており、その枠組みの中で「何をすべきか」をユーザコードに問い合わせる。そこでは、もはやユーザコードはアプリの管理者ではなく、フレームワークの要求に従うだけの個々の歯車にすぎない。

  このようにプログラム実行の主体が逆転することを**制御の反転（IoC：Inversion of Control）** という。本来、フレームワークとライブラリとは、IoCの性質によって使い分けるべきキーワードである。もっとも、実際の開発現場では区別されないで使われることがあることも事実である。
